## Design Principles

| Principle | Implementation |
|-----------|---------------|
| **FK everywhere** | All relations have explicit FK constraints with ON DELETE action |
| **Soft delete** | `deleted_at TIMESTAMPTZ` on `profiles` and `listings`; RLS filters these rows |
| **Proper enums** | 6 enum types; no magic strings in status columns |
| **PostGIS** | `GEOGRAPHY(POINT, 4326)` on `profiles` and `listings`; GiST indexes |
| **RLS on all tables** | Row Level Security enabled; public/own/participant policies per table |
| **Audit timestamps** | `created_at`, `updated_at` on mutable tables; `set_updated_at()` trigger |
| **Server-side secrets** | `pickup_code` generated by `generate_pickup_code()` PG function ‚Äî never passed from client |

## Soft Delete Pattern

`profiles` and `listings` are never hard-deleted:

```sql
-- Soft-delete a listing
UPDATE listings SET deleted_at = NOW(), status = 'archived' WHERE id = $1;

-- Soft-delete a profile (GDPR request)
UPDATE profiles SET deleted_at = NOW(), name = '[deleted]', avatar_url = NULL, bio = NULL
WHERE id = $1;
```

RLS policies include `deleted_at IS NULL` in the `USING` clause for SELECT.

## Realtime Subscriptions

Two tables are published to `supabase_realtime`:

| Table | Use case |
|-------|---------|
| `messages` | Live chat (per-conversation filter recommended) |
| `notifications` | Unread badge, notification feed |

**Recommended filter pattern for messages:**
```typescript
supabase.channel(`chat:${conversationId}`)
  .on('postgres_changes', {
    event: 'INSERT', schema: 'public', table: 'messages',
    filter: `conversation_id=eq.${conversationId}`
  }, handler)
```

**Recommended filter pattern for notifications:**
```typescript
supabase.channel('my-notifications')
  .on('postgres_changes', {
    event: 'INSERT', schema: 'public', table: 'notifications',
    filter: `user_id=eq.${userId}`
  }, handler)
```

## Listing Expiry

Listings auto-expire 30 days after creation via `expires_at`. A scheduled job (Supabase Edge Function or cron) should:

```sql
-- Cron: daily at 3am UTC
UPDATE listings
SET status = 'archived', updated_at = NOW()
WHERE status = 'available'
  AND expires_at < NOW()
  AND deleted_at IS NULL;
```

Owner receives `listing_expiring` notification 3 days before expiry.

## Differences vs MVP Schema (lil-peep)

| Area | MVP | v2 |
|------|-----|----|
| Reservations | Basic (4 states) | Full flow (7 states + timestamps) |
| Reviews | FK to `listing_id` only | FK to `reservation_id` (exchange required) |
| Pickup code | Passed from client | Generated server-side in `confirm_reservation` |
| Listings | No expiry, no condition, no soft-delete | `expires_at`, `condition`, `deleted_at` |
| Notifications | 4 types, no `action_url`/`related_id` | 10 types, `action_url`, `related_id`, `read_at` |
| Conversations | No reservation link | `reservation_id` FK |
| Messages | No system messages | `is_system` field |
| Favorites | Missing | Added |
| Realtime channels | No user-level filter | User-level filter in notification queries |
| Functions | `create_reservation` (client passes code) | Full RPC suite ‚Äî all state transitions |
